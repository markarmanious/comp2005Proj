Use Cases and User Stories

Account Lockout User Story

The user attempts to login to her account. The user enters the wrong password. The user makes several attempts to log in again with the wrong password. After several failed attempts, the SuD locks the account associated with the user name.

Use Cases

Title: Account Registration
Primary Actor: New User
Stakeholders and Interests:

User - wants to create an account.
Website Owner - wants to let users access the system.
Precondition: User doesn't have an account.
Main success scenario:
  1. User clicks on login button.
  2. SuD redirects to login page.
  3. User clicks 'create account'.
  4. SuD redirects user to account creation page.
  5. User enters a username and password.
  6. User account is successfully created.
  7. User is automatically logged in.
Extensions:
6a. User account already exists:
6a1. SuD notifies user that the username is already in use
6a2. SuD prompts user to either enter a different username or click 'forgot password'.
6a3. User returns to step 5 or is redirected to password recovery.


Title: Posting a Topic Thread
Primary Actor: Poster
Stakeholders and Interests:

Poster - wants to make a new topic thread and post in it.
Other users - want to see the new topic thread and posts.
Precondition: Poster is logged into their account.
Main success scenario:
  1. Poster logs in.
  2. Poster navigates to topics page.
  3. Poster clicks 'create new topic'.
  4. Poster enters topic title and initial post.
  5. SuD processes submission and displays new topic with other topics.
  6. Other users access new topic.
  7. Other users post in new topic.
Extensions:
5a. Topic already exists:
5a1. SuD notifies user that the topic already exists.


Title: Receive Subscription Notification
Primary Actor: User
Stakeholders and Interests:

Subscribed User - wants to receive notifications about new posts in a subscribed thread.
SuD - wants to notify all subscribed users that a new post has been made in the thread.
Poster - wants other users to be aware of their new post.
Precondition: User is subscribed to the topic thread.
Main success scenario:
  1. Poster makes a new post in a topic thread which has subscribers.
  2. Subscribed user logs in to her account to view topics.
  3. SuD recognizes that a new post has been made in a subscribed topic thread which the user has not seen.
  4. SuD notifies user of the new post.
  5. User views the topic with the new post.
  6. SuD removes notification of new post.
Extensions:
5a. User does not view the new post:
5a1. Notification remains until the post is viewed.


Title: Viewing a Restricted Topic
Primary Actor: User
Stakeholders and Interests:

User - wants to view a topic restricted to a specific discussion group.
Group - wants to ensure only group members can view the topic.
Precondition: User is logged into their account and belongs to a discussion group.
Main success scenario:
  1. User clicks on a topic that is restricted to a discussion group.
  2. SuD loads the group membership from the database.
  3. SuD compares the user ID to the group member IDs.
  4. SuD finds the user ID in the group membership.
  5. SuD loads the topic page.
  6. User views the topic page.
  7. User posts in the topic page.
Extensions:
4a. SuD does not find the user ID in the group membership:
4a1. SuD notifies user that the topic is restricted to a group of which they are not a member.
4a2. User returns to main topics page.


Title: Post a Tutorial Thread
Primary Actor: Poster
Stakeholders and Interests:

Poster - wants to create a new thread to provide information to other users.
User - wants to be able to access this information and reply to the thread.
Moderator - wants to ensure the content follows the rules of the forum.
Precondition: Poster has an account and is logged in.
Main success scenario:
  1. Poster selects the appropriate subforum for the topic of their tutorial.
  2. SuD navigates to this subforum.
  3. Poster chooses to create a new thread.
  4. SuD opens a new thread page with text boxes for the title, body, and optional tags.
  5. Poster enters a title for the tutorial.
  6. Poster enters the body of the tutorial.
  7. Poster enters tags for the tutorial to be retrieved by the search function.
  8. Poster submits the thread.
  9. SuD sends the thread to the moderator for revision.
  10. Moderator approves the thread and posts it to the forum.
Extensions:
10a. Moderator rejects thread:
10a1. Moderator provides reason for rejection.
10a2. Poster instantiates Modify Posting use case.


Title: Account Recovery
Primary Actor: User
Stakeholders and Interests:

User - wants to recover their password
Precondition: User has forgotten their password and knows their security question/answer.
Main success scenario:
  1. User fails login attempt.
  2. User clicks forgot password link.
  3. User is prompted to answer a security question.
  4. User answers security question.
  5. User resets password.
  6. User accesses account.
Extensions:
4a. User provides incorrect answer:
4a1. User may try again a limited number of times before being locked out of password reset.
